# 全栈架构师的思维基石：如何从零推导并规划数据类型

很多新手在开发时习惯“边写边凑”，需要什么属性就在代码里现加。这会导致代码极度脆弱。
高级全栈工程师在写下第一行逻辑代码前，一定会在脑海里进行**“数据类型推演”**。

这其实并不难，你不需要凭空想象，只需要针对每个新功能，刻意问自己**三个核心问题（数据流的三级契约）**：

---

## 契约一：底层数据长啥样？（DB Schema）
**思考目标**：为了让这个功能永久保存，数据库里最起码得有哪几个柱子支撑？
这是最容易想到的。

**【实战举例】假设你要给 IdeaFlow 加一个“文章评论（Comment）”功能：**
你闭上眼睛想，一条评论最起码得有什么？
1. 谁发的？(`userId`)
2. 发在哪篇文章下？(`articleId`)
3. 说了什么？(`content`)
4. 什么时候发的？(`createdAt`)

*好，第一层数据库的 Type 就出来了！*

---

## 契约二：用户发请求时，必须填什么表？（Request Schema）
**思考目标**：前端要想办成这件事，必须（且只需）传给我什么数据？
这个类型通常比数据库字段要**少**，因为它是由前端提交的。

**【推导逻辑】继续看评论功能：**
用户在前端点“发表评论”时，前端的请求体需要包含刚才数据库里的所有字段吗？
- `userId`：不需要传！后端可以从他的登录 Cookie/Token 里自己解析出来。前端如果传了这个，反而不安全（容易被伪造）。
- `createdAt`：不需要传！后端接收到请求的一瞬间，自己取当前时间 `new Date()` 就行了。
- `articleId`：**需要！** 我得知道他是在哪篇文章下评论的。
- `content`：**需要！** 我得知道他说了啥。

**结论**：用户发评论的 Zod Schema 就推导出来了：
```typescript
export const createCommentSchema = z.object({
    articleId: z.string().uuid("必须是有效的文章ID"),
    content: z.string().min(1, "评论总不能是空的吧？").max(500, "不能长篇大论"),
});
```

---

## 契约三：我办完事，要给前端返回什么？（Response Schema）
**思考目标**：前端拿到了我的返回数据，要在页面上画 UI。他画 UI 需要什么，我就给他拼什么。
这个类型通常比数据库字段要**多**或者**经过修饰**。

**【推导逻辑】前端渲染评论列表，需要什么？**
数据库里只有：`userId`, `articleId`, `content`, `createdAt`。
前端只拿到 `userId` 画不出东西，前端需要展示用户的**头像**和**昵称**！
所以，后端在查出评论后，还得去用户表里把头像和名字查出来拼上去。

**结论**：评论列表的返回 Zod Schema 推导出来了：
```typescript
export const commentResponseSchema = z.object({
    id: z.string(),
    content: z.string(),
    createdAt: z.string(), // 把后端的 Date 对象格式化成字符串发给前端
    author: z.object({
        name: z.string(),
        avatar: z.string().url()
    }) // 这是拼装出来的数据，而不是原生数据库字段
});
```

---

## 核心总结

不要觉得“一开始写 Zod 很难想”。你之所以觉得难想，是因为你想一口气把前端、后端、数据库需要的类型揉在一起思考。

把你的思维拆分成这三步流水线：
1. **存数据库的原始原料是什么？（DB Schema）**
2. **前端发请求的最小必要条件是什么？（Request Schema）**
3. **前端画页面的最终成品是什么？（Response Schema）**

每次写新功能，先建个 `.ts` 文件，强迫自己先把这三个 Zod 对象写出来。
写完这三个东西，你会发现，中间的业务逻辑（Service层），无非就是：**把 Request 转换成 DB，再把 DB 转换成 Response 的过程而已！** 
此时，你的代码思路将势如破竹。
