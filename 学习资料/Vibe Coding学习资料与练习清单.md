# 📚 Vibe Coding 学习资料与练习清单

> 学习资料最重要的作用，不是让你知道更多术语，而是：
> **帮助你练习读需求、问需求、拆需求、复盘需求。**

---

## 🧭 一、学习资料的使用原则

你现在最需要的，不是无限搜资料，而是建立一个能持续训练的输入源。

### 学习资料优先围绕这 4 类来找

1. 真实产品
2. 真实工程讨论
3. 真实需求表达
4. 你自己的项目实践

### 一句话理解

> 最好的资料，不是“某门万能课程”，而是能让你练习需求理解和需求拆解的材料。

---

## 🔍 二、最值得看的资料来源

### 1. GitHub 开源项目的 Issues / PR / Discussions

这是最接近真实工程需求的地方。

#### 推荐你重点关注的项目类型
- AI Agent / workflow
- AI app / chat / retrieval
- 前端全栈应用
- 内容工作流 / 知识库工具

#### 可重点看的项目
- `vercel/ai`
- `langchain`
- `langgraph`
- `dify`
- `flowise`
- `open-webui`
- `lobehub`
- `cline`
- `openhands`

#### 看什么？
- issue 是怎么描述问题的
- 评论里别人如何 уточ清需求
- 最终 PR 改了哪几层
- release note 怎么总结改动

#### 训练方式
每次选一个 issue，自己先写：

- 这个问题的真实用户痛点是什么？
- 真正需求是什么？
- 我会拆成哪些任务？

再去看别人怎么做。

---

### 2. 产品官网 / Demo 视频 / 发布视频

你以后很多需求总结能力，会来自“看懂一个产品想解决什么问题”。

#### 推荐关注的产品方向
- AI 写作助手
- AI 搜索
- AI 编程助手
- AI 面试 / 求职工具
- 知识管理工具
- Agent 工作流平台

#### 推荐产品
- Notion AI
- Perplexity
- Cursor
- Replit AI
- Granola
- v0
- Bolt.new
- Dify
- Flowise
- Cline / Windsurf

#### 训练方式
看完一个产品 demo 后，自己回答：

1. 核心用户是谁？
2. 核心问题是什么？
3. MVP 是什么？
4. 这个功能为什么值得做？
5. 如果我来做第一版，我会先做哪 3 个点？

---

### 3. 招聘 JD

这是最被低估的学习材料之一。

#### 为什么看 JD 很有用？
因为 JD 本质上就在告诉你：

- 企业需要什么能力组合
- 他们怎么描述业务问题
- 他们要你解决什么类型的问题

#### 看 JD 时重点提取什么？
- 岗位真正看重的是前端、后端、还是产品落地
- AI 相关要求偏：模型接入 / workflow / 检索 / 多轮对话 / 自动化
- 他们写的“快速迭代 / vibe coding / owner 意识”到底落在哪些能力上

#### 训练方式
每看一个 JD，写一段：

- 这家公司最缺的人是什么样？
- 这个岗位最核心的问题是什么？
- 我现在哪些项目点能对上？
- 我缺哪些能力？

---

### 4. 产品 / 创业 / 工程复盘内容

看别人怎么讲“为什么做这个功能”，比看纯教学视频更值钱。

#### 可以看的平台
- YouTube
- B站
- X / Twitter
- 即刻
- 少数派
- 独立开发者博客
- 开发者 Newsletter

#### 重点关注的内容类型
- 产品设计复盘
- 功能迭代复盘
- AI 应用构建过程
- 从需求到 MVP 的思路
- 为什么这个功能不做那个版本

#### 训练目标
学会这种表达：

- 为什么做
- 给谁做
- 先做什么
- 不做什么
- 技术如何支撑产品目标

## 🧪 三、你自己的项目也是学习资料

你现在已经有项目了，所以最有价值的训练场景，其实是你自己的项目迭代。

以后每次想加功能，不要直接开干，先写一张需求卡片。

### 需求卡片模板

```text
功能名：
目标用户：
使用场景：
用户问题：
功能目标：
输入：
输出：
最小 MVP：
暂时不做：
技术影响：
```

### 你可以专门建一个文件夹记录这些卡片
比如：

- `学习资料/需求卡片/AI文章助手.md`
- `学习资料/需求卡片/语义搜索.md`
- `学习资料/需求卡片/面试模拟追问.md`

这个习惯会极大提高你的需求总结能力。

## 📅 四、每周训练计划（建议版）

### 周训练目标
每周做 3 次训练，每次 30-45 分钟。

---

### 训练 A：拆一个产品

输出模板：

```text
产品名：
目标用户：
核心问题：
核心功能：
MVP：
技术猜测：
我会怎么做第一版：
```

---

### 训练 B：拆一个 GitHub issue

输出模板：

```text
项目名：
Issue 链接：
真实问题：
用户痛点：
影响范围：
我会拆的任务：
对照原作者实现后的收获：
```

---

### 训练 C：为你自己的项目写功能需求卡

输出模板：

```text
功能名：
为什么值得做：
最小 MVP：
前端要改什么：
后端要改什么：
AI 应该参与哪一步：
风险点：
```

---

### 训练 D：口头表达（加分项）

录 3 分钟音频，回答：

- 这个功能解决谁的问题？
- 为什么先做 MVP？
- 为什么不用另一个方案？
- 我打算先实现哪一部分？

这一步对面试帮助非常大。

---

## 🪜 五、最适合你的资料使用顺序

### 第一阶段：建立感觉（1-2 周）
重点看：
- AI 产品 demo
- 招聘 JD
- 少量产品复盘内容

#### 目标
- 学会判断“一个产品到底在解决什么问题”

---

### 第二阶段：工程训练（2-4 周）
重点看：
- GitHub issues / PR / discussions
- 自己项目的功能需求卡

#### 目标
- 学会拆任务、拆层次、拆 MVP

---

### 第三阶段：表达和复盘（长期）
重点做：
- 口头 3 分钟表达
- 写功能复盘
- 写项目演进记录

#### 目标
- 让你不只是会做，还能讲得出来

---

## ✅ 六、判断资料是否值得看的标准

以后你看到一个资料，先问自己：

### 这份资料能不能帮助我回答下面至少一个问题？
- 用户真正的问题是什么？
- 功能为什么值得做？
- 最小版本是什么？
- 怎么拆成前后端任务？
- 为什么选这个技术方案？

如果不能，说明它大概率只是“信息娱乐”，不是高质量训练材料。

---

## ⚠️ 七、你最该避免的无效学习方式

### 1. 只看“如何实现”而不问“为什么实现”
这会让你越来越像代码搬运工。

### 2. 一上来就找完整课程，结果一直收藏不练
你的阶段更适合“小输入 + 高频练习”，不是沉迷系统课。

### 3. 只学新框架，不练需求拆解
这会让你技术名词越来越多，但产品理解没长。

### 4. 一看到 AI 就只想做聊天框
这对 AI Agent 方向帮助不大，真正值钱的是 workflow、检索、状态、工具调用。

---

## 🚀 八、推荐你现在就开始做的事

### 立刻可做的 3 件事

1. 每周拆 1 个 AI 产品
2. 每周拆 1 个 GitHub issue
3. 给你当前项目每个新功能写需求卡

如果你能连续做 4 周，你会明显感觉：

- 需求不再那么模糊
- 功能优先级更清楚
- 跟别人讨论项目时更有条理
- 面试里更容易答出“我为什么这么做”

---

## 🧠 最后一句总结

真正有用的学习资料，不是让你知道更多概念，而是：

> **让你学会把模糊的想法，变成可以落地的产品需求和工程任务。**

你以后挑资料，就按这个标准来。
