# 🧠 独立开发者的编程心智模型

> **写给没有商业项目经验的你**：拿到一个需求后，如何从"大脑一片空白"变成"手指在键盘上飞舞"。

---

## 第一章：功能拆解四步法（万能公式）

遇到任何新功能，强迫自己按这 4 步走。**不要跳步，不要靠灵感。**

### 第 1 步：数据长什么样？（Database / Schema）

> 核心问题：**"这个功能需要存什么数据？字段叫什么？类型是什么？"**

**举例 — 回收站功能：**

- 回收站里的文章 = 普通文章 + 一个"被删除时间"的标记
- 所以只需要在 `articles` 表里加一个 `deleteAt` 字段
- 有值 = 在垃圾桶里；`null` = 正常文章

**举例 — 点赞功能：**

- 需要知道"谁"给"哪篇文章"点了赞
- 所以需要一张新表：`likes`，字段：`userId`, `articleId`, `createdAt`

**口诀：先想"存什么"，再想"怎么写代码"。**

#### ⚠️ 补充：前端的"数据"不是数据库！

四步法的第 1 步在不同阶段，"数据"含义不同：
- **写后端时**："数据"= 数据库字段（永久保存的，如 `deleteAt`）
- **写前端组件时**："数据"= `useState`（临时的，页面关了就没了）

前端怎么想出需要哪些 `useState`？问自己：**"这个组件里什么东西会变化？"**
```
搜索弹窗里什么会变？
1. 输入框的文字会变    → useState 存关键词
2. 复选框会被勾选/取消 → useState 存开关状态
3. 搜索结果列表会变    → useState 存结果数组
```
**凡是界面上会变化的东西，就需要一个 `useState`。**

---

### 第 2 步：后端提供什么弹药？（API / Service）

> 核心问题：**"前端页面需要后端提供哪些'动作'？"**

把自己想象成一个餐厅服务员，你站在厨房门口（后端），前台客人（前端）会对你喊什么？

**举例 — 回收站功能：**

1. "给我看看垃圾桶里有什么" → `GET /api/articles/trash`
2. "把这篇文章从垃圾桶捞回来" → `PATCH /api/articles/:id/restore`
3. "彻底销毁这篇文章" → `DELETE /api/articles/:id` (硬删除)

**举例 — 点赞功能：**

1. "这篇文章有多少赞？" → `GET /api/articles/:id/likes`
2. "我要点赞/取消点赞" → `POST /api/articles/:id/like`

**口诀：一个用户操作 = 一个 API 接口。**

---

### 第 3 步：用户看到什么？（UI 界面）

> 核心问题：**"用户在哪里触发这个功能？点完之后看到什么？"**

不用画设计稿，用文字描述就行：

**举例 — 回收站功能：**

- 侧边栏底部有一个 🗑️ 回收站按钮
- 点击后弹出一个列表弹窗（或者跳转到 `/trash` 页面）
- 列表里每一行显示：文章标题 + 删除时间
- 每一行右边有两个按钮："恢复" 和 "彻底删除"

**口诀：先用"嘴"把界面说出来，再用代码画出来。**

#### 🧩 补充：组件需要什么参数（Props）？

问自己：**"这个组件自己搞不定什么事？需要外面帮忙什么？"**
- 自己能做的（调 API、管状态）→ 不需要 prop，自己内部搞定
- 做不了的（关闭弹窗、父组件传来的数据）→ 需要从外面传进来

**举例 — SearchModal：**
- 搜索数据 → 自己调 API 拿 → 不需要 prop
- 关闭弹窗 → 控制权在父组件（Sidebar）→ 需要 `onClose` prop

**口诀：自己能做的自己做，做不了的让外面传。**

#### 🔌 补充：怎么知道要 import 什么？

不需要一开始就写好所有 import！正确的流程：
1. 先写组件壳子（函数 + return 一个 div）
2. 写到需要 `useState` → IDE 报红 → 去顶部加 import
3. 写到需要图标 → IDE 报红 → 去顶部加 import
4. 写到需要跳转 → IDE 报红 → 去顶部加 import

**口诀：边写边引入，用到什么就 import 什么。去项目里已有的类似文件"偷"import 语句。**

---

### 第 4 步：点击后发生什么？（交互逻辑 / Hooks）

> 核心问题：**"用户点了按钮后，代码要做哪几件事？顺序是什么？"**

**举例 — 点击"恢复"按钮：**

1. 调用 `client.api.articles[":id"].restore.$patch({ param: { id } })`
2. 如果成功 → 弹出 Toast："文章已恢复"
3. 刷新当前列表（`router.refresh()`）
4. 如果失败 → 弹出 Toast："恢复失败，请重试"

**口诀：一个按钮 = 一个异步函数 = 调接口 → 判断成功/失败 → 更新界面。**

## 第二章：Debug 排错方法论（救命指南）

代码跑不起来时，不要慌，不要乱改。按照这个**"从外到内"**的顺序排查：

### 🔴 第 1 层：看浏览器报错（Console）

打开浏览器 → 按 `F12` → 点 `Console` 标签页。

- **红色错误 (Error)**：通常会告诉你哪个文件、第几行出了问题。
- **黄色警告 (Warning)**：暂时不管，不影响功能。

### 🟠 第 2 层：看网络请求（Network）

还是 `F12` → 点 `Network` 标签页 → 找到你刚才触发的那个请求。

- **状态码 200**：后端没问题，问题在前端。
- **状态码 404**：你的 URL 路径写错了。
- **状态码 500**：后端炸了，去看终端的服务器日志。

### 🟡 第 3 层：看终端日志（Terminal）

回到你跑 `pnpm dev` 的那个终端窗口。

- 如果有红色报错，通常会精确到文件名和行号。
- 常见原因：`import` 路径写错、变量名拼写错误、忘记 `export`。

### 🟢 第 4 层：加 `console.log` 定位

如果前三层都找不到原因，就在你怀疑的代码附近到处插 `console.log`：

```typescript
console.log("1. 函数进来了");
console.log("2. title 的值是:", title);
console.log("3. API 返回的数据:", data);
```

通过观察哪些 log 打印了、哪些没打印，你就能精确定位到"代码在哪一行卡住了"。

### 💀 终极手段：回滚（Git）

如果你改了一大堆代码，已经不知道自己改坏了什么：

```bash
git stash        # 把当前所有修改暂存起来（不会丢失）
git stash pop    # 想恢复时再拿回来
```

或者更狠：

```bash
git checkout -- .  # 放弃所有未提交的修改，回到上次 commit 的状态
```

## 第三章：三层架构思维（代码往哪放？）

当你写完一个功能，发现代码全堆在一个文件里，又臭又长。这时候你需要掌握"分层"的思想。

### 🏗️ 三层架构（你正在用的）

```
用户点击按钮
    ↓
【Controller 层】api.ts / route.ts
    职责：接收请求，校验参数，返回响应
    口诀："我只负责接电话和挂电话"
    ↓
【Service 层】service.ts
    职责：处理业务逻辑（判断文章存不存在、软删除还是硬删除）
    口诀："我是干活的主力"
    ↓
【Data 层】schema.ts + Drizzle ORM
    职责：直接跟数据库对话
    口诀："我只管存和取"
```

### 为什么要分层？

**不分层的后果：** 你把数据库查询、业务判断、错误处理全写在一个路由函数里。

- 改一个小逻辑，要在 200 行代码里翻来翻去。
- 想复用"查文章"的逻辑？只能复制粘贴，改一处忘改另一处。

**分层后的好处：** 你的 `api.ts` 只有 5-10 行调用代码，干净得像一份菜单。真正的逻辑全在 `service.ts` 里，想改就改一处，全局生效。

## 第四章：代码组织口诀（文件往哪放？）

### 口诀 1：按"功能模块"分文件夹，不按"文件类型"分

```
✅ 推荐（按功能）：
src/
├── server/
│   └── articles/        ← 文章相关的全在这
│       ├── api.ts
│       ├── service.ts
│       └── schema.ts
│   └── users/           ← 以后用户相关的全在这
│       ├── api.ts
│       ├── service.ts
│       └── schema.ts

❌ 不推荐（按类型）：
src/
├── controllers/         ← 所有 controller 堆一起
├── services/            ← 所有 service 堆一起
├── schemas/             ← 所有 schema 堆一起
```

### 口诀 2：重复出现 3 次就抽离

- 同一个 `interface Article` 在 3 个文件里出现了？→ 抽到 `types/article.ts`
- 同一段"格式化日期"的代码写了 3 遍？→ 抽到 `lib/utils.ts`
- 同一个"防抖保存"逻辑要在多个编辑器用？→ 抽成一个自定义 Hook

### 口诀 3：命名要让"半年后的自己"看得懂

```typescript
// ❌ 糟糕的命名
const d = await db.select()...        // d 是什么？
const handleClick = async () => ...   // 点击了干嘛？
function process(x) { ... }           // 处理什么？

// ✅ 清晰的命名
const articleList = await db.select()...
const handleDeleteArticle = async () => ...
function formatDateToChineseString(date) { ... }
```

## 第五章：心态管理（最重要的一章）

### 🧊 冷静法则：改代码之前，先深呼吸

> **"Bug 不会因为你着急就自己消失。"**

当你遇到报错时：

1. **不要**疯狂地到处改代码碰运气。
2. **要**先花 30 秒读完报错信息。80% 的报错信息会直接告诉你答案。

### 📐 最小验证法则：一次只改一个地方

> **"如果你同时改了 5 个地方，然后代码跑通了，你根本不知道是哪个改动起了作用。"**

正确的做法：改一处 → 保存 → 刷新 → 看效果 → 再改下一处。

### 🎯 完成大于完美

> **"先让它跑起来（Make it work），再让它变好看（Make it right），最后让它变快（Make it fast）。"**

很多初学者（包括你）容易陷入一个陷阱：

- 功能还没写完，就开始纠结变量名好不好、代码够不够优雅。
- 结果一天过去了，功能一个没上线。

**正确的节奏：**

1. 第一遍：用最丑的代码把功能跑通（哪怕全写在一个函数里）。
2. 第二遍：重构代码结构（分层、抽函数、改命名）。
3. 第三遍：优化性能（加缓存、减少请求次数）。

### 🔄 不要怕重写

> **"你写的第一版代码，注定会被你自己推翻。这不是失败，这叫成长。"**

你现在的 `useArticleEditor.ts` 已经被我们重构了至少 3 次。
每一次重构，你都比上一次理解得更深。这就是编程学习的本质。

---

> **💡 终极总结**
> 编程不是"灵光一现"的艺术，而是"按部就班"的工程。
> 当你拿到一个需求时，不要盯着空白的编辑器发呆。
> 打开这份文档，从第 1 步开始：**"这个功能需要存什么数据？"**
> 然后一步一步往下走。走完四步，代码自然就出来了。
