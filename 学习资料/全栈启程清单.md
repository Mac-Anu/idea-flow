## 🗺️ 全视角系统防御图鉴
> 这是你亲自搭建的大厂级数据处理水路系统。顺着箭头看，你会发现哪怕是一个简单的登录请求，也经历了层层绝对安全的重型关卡。

```mermaid

    subgraph 后方核聚变堡垒 ⚙️
        Middleware[Hono Middleware 强类型安检机]

    subgraph 持去化基岩 🗄️
        ORM[Drizzle ORM]

    Form ==>|收集完毕,下达请求| Wrapper
```

## 🏗️ 第一阶段：地基搭建与视觉规范（前端物理层）

这是任何项目启程的绝对第一步：决定系统长什么样，以及页面的物理结构如何组织。

### 1. Next.js App Router 目录心法
- **动静分离 (`page.tsx` vs 独立组件)**：凡是不需要复杂交互的壳子（如页面大标题、背景），一律写成服务器渲染的 `page.tsx`（极致SEO与加载速度）；凡是需要点击、输入、弹窗的“交互岛屿”（如登录表单），一律单独抽成带有 `"use client"` 的组件。
- **路由黑魔法**：
  - 用 `(pages)` 等带括号的路由分组，可以让不同页面共享不同的 `layout.tsx` 侧边栏，但不弄脏 URL。
  - 用 `_components` 下划线文件夹可以在页面文件夹里藏各种私有组件，防止意外变成网页被人看到。

### 1.3 从“按类型”向“按领域 (DDD)”觉醒（项目目录架构）
- **遗弃的旧习惯**：把全服所有的钩子扔进一个最大 `hooks` 文件夹。
- **大厂新思维 (Feature Slice / 领域驱动目录架构)**：这是大型项目不崩溃的核心基石。把 `Login的表单`、`Login的接口` 和 `Auth的类型定义` 连锅端，打包装车为一个叫 `src/features/auth/` 的军团要塞。每一次抽离与分拆，都是为了防止十万行代码后的系统雪崩。

**当前的实际物理架构图鉴（持续进化中）：**
```text
src/
 ├─ app/                 (雷打不动！仅保留各个网页的壳子)
 │   ├─ (auth)/          (🔐 路由组：无边栏的纯净阵营)
 │   │   ├─ layout.tsx      (空白居中布局)
 │   │   └─ login/page.tsx  (登录页面的静态绝缘壳)
 │   │
 │   └─ (main)/          (🏠 路由组：带侧边栏的主力阵营)
 │       ├─ layout.tsx      (包含 Sidebar 的全局布局)
 │       └─ page.tsx        (项目首页)

     ├─ common/          (大门护卫队：如 error/app 创建工厂)
     ├─ user/            (用户后端防区)
     │   ├─ schema.ts       (表结构与 Zod 天书规则)
     │   ├─ service.ts      (处理具体的加解密业务脏活)
     │   ├─ middlwares.ts   (拦截黑客的 Hono 身份安检门！)
```

### 2. 打造极客工业级 UI（Tailwind + Shadcn）
- **去工具化原则**：放弃了重得要死的传统 UI 库，使用了 `shadcn/ui`。我们将原始的输入框、按钮直接沉淀到了 `src/components/ui/`，随时可以更改其底层圆角与呼吸灯特效。
- **Notion 极简美学**：在全局的 `globals.css` 中注入了 CSS 变量体系，全系统严格遵守极简黑暗/明亮色系，再也不需要去组件里零散拼凑 `bg-red-500` 这样的游击代码。

## 🗄️ 第二阶段：血肉与骨骼模型（数据库持久层）

界面画完，下一步永远是想“我们要存什么数据”。必须先有数据，再有业务逻辑。

### 1. 结构化你的世界（Drizzle ORM & Schema）
- **唯一真理之源**：我们在 `schema.ts` 里严格划定了文章存什么、用户存什么类型的字段（比如 `deleteAt` 代表回收站）。摒弃了手动拼写 SQL 字符串的野蛮时代，用 Drizzle ORM 保障每一道查询都有极强的 TypeScript 补全提示。

### 2. 面向未来的三层架构（业务架构核心）
当你再也不愿意看一坨长达 500 行的乱七八糟代码时，你顿悟了它的价值：
- **Data（数据层）**：即 ORM 的 Schema，只管表结构设计。
- **Service（干活主力）**：比如处理“判断文章存不存在、是软删除还是硬删除”的业务逻辑大脑（如：`src/server/articles/service.ts`）。
- **Controller/Route（接线员）**：只负责大门口接客、校验参数错没错，然后丢给 Service 干活（如：`src/server/articles/route.ts`）。

## 🌉 第三阶段：打通全栈经脉（API 桥梁层）

当前沿战壕（前端）和后方堡垒（数据库）建好后，你必须建立一条极其坚固、绝对安全的防静电专线。

### 1. 统一调度的独裁者：配置大总管 (`appConfig.ts`)
- 我们杜绝了在全网几十个文件里到处乱写 `http://localhost:3000` 或 `process.env` 的噩梦。所有环境地址统一在配置中心收口，换域名、换服务器、上架到全世界，只需改这一个文件，天下百万雄师立刻原地转身！

### 2. 全连接与防腐层（Hono + 包装法）
- **Hono RPG 连桥**：后端写出的路由接口类型，前端直接无缝吸取，改个字段名双端同时引爆警报。
- **两层包装法 (`src/api/*.ts`)**：前端不直接去摸诸如 `better-auth` 或者后端的原始请求，而是跨越建立一个如 `authApi.signIn` 自己的专线盒子。如果日后第三方库大洗牌，由于你隔了一层“防腐层”，网页端的逻辑只需微微一笑，一行代码不用改。

## 🛡️ 第四阶段：全城最高安保（认证防线层）

此时你的系统已经能跑了，但大门洞开。我们要装上视网膜识别机和防暴盾。

### 1. 全端统一校验（Single Source of Truth)
- 最顶级的偷懒术：后端立下 `loginRequestSchema` (Zod 天书规则：密码必须大于六位)。前端借用 `zodResolver` 直接隔空拉取这一份天书！
- **效果**：不仅用户在网页上输错会瞬间亮红灯提示，哪怕黑客从底层偷偷塞垃圾数据，前后两扇门共用一套密码体系，谁也进不来。绝不用在两端手写两份验证代码！

### 2. 隐形保护圈：造台收音机 (React Context)
- 为了解决组件传值极其冗长的 Prop Drilling 灾难。我们在云端架起了 `AuthContext` 电台。
- **大喇叭 (`useSetAuth`)**：登录成功，向全网吼一嗓子“注入当前身份！”。
- **窃听器 (`useAuth`)**：导航栏、文章评论区偷偷探听电台。如果是游客，就只渲染登录按钮；如果是主人，就渲染删除按钮（这叫大厂最核心的**“前端组件保护保护”**！）。

### 3. 终极防空火力网：带装甲的中间件 (Hono Middleware)
- **底层死守**：虽然网页上的“删除按钮”对游客隐藏了，但黑客可以直接发请求删库。所以我们在后端的某些敏感大门外强行加了一扇安检机。
- **没有通行证 (`getSession`)** 瞬间当头一炮 `401 Unauthorized` 轰出服务器。
- **智能装甲 (`createMiddleware`)**：既然在不同文件切分逻辑，普通代码早就把 TS 的智能提示切断了。但我们启动了 Hono 的重兵工厂 `createMiddleware`：你哪怕跨越八千个文件，在下流拿到的依然是活生生、能 `.` 出属性名的绝密身份档案！
